## Context

Shorty currently has no authentication — any client that can reach the service can create URLs via `POST /Create-URL`. The target deployment places Shorty behind an identity-aware reverse proxy (e.g., Cloudflare Zero Trust, Azure App Proxy) that authenticates users against Microsoft Entra ID and injects the verified email into the `X-Authenticated-User` header.

This design adds a FastAPI dependency that reads the header AND verifies an HMAC-SHA256 signature proving the header was generated by the proxy (not forged by a client). Both proxy and Shorty share a secret (`PROXY_SHARED_SECRET`). The backend does not manage sessions, issue JWTs, or talk to Entra ID directly.

The project follows a hexagonal layout:
```
src/api/       → routes, schemas, dependencies
src/core/      → use cases, ports, domain types
src/infra/db/  → SQLAlchemy models, repository
```

## Goals / Non-Goals

**Goals:**
- Add a reusable `RequireAuthenticatedUser` FastAPI dependency that extracts the `X-Authenticated-User` header, verifies its HMAC-SHA256 signature against `X-Auth-Signature` using a shared secret, and returns the email or raises `403 Forbidden`.
- Apply the dependency to `POST /Create-URL` so only authenticated users can create URLs.
- Persist the authenticated user's email in a new `created_by` column on the `urls` table.
- Keep `GET /{short_code}` fully public — no auth, no header required.

**Non-Goals:**
- No JWT/session management in the backend.
- No Entra ID integration code (proxy handles it).
- No user registration, login UI, or password flow.
- No configuration of the reverse proxy or Entra ID tenant.
- No `PUT`/`DELETE`/listing endpoints for URLs (deferred to a future change).
- No per-user rate limiting or TTL (separate changes).

## Decisions

### HMAC signature verification

Header-only auth is vulnerable to spoofing — a client that bypasses the proxy can inject any `X-Authenticated-User` value. To prevent this, the proxy signs the email with an HMAC-SHA256 using a shared secret, and Shorty verifies the signature before accepting the header.

**Flow:**

```
Proxy:  sig = base64(HMAC-SHA256(secret, email))
        → X-Authenticated-User: jperez@empresa.com
        → X-Auth-Signature:    a1b2c3d4e5f6...

Shorty: expected = base64(HMAC-SHA256(shared_secret, email))
        if expected != X-Auth-Signature → 403
```

The secret lives in `Settings`:

```python
proxy_shared_secret: str = Field(
    default="",
    description="Shared HMAC secret for verifying X-Auth-Signature from the proxy.",
)
```

A blank default means no HMAC verification — useful for local development without a proxy. In production, `PROXY_SHARED_SECRET` MUST be set and identical on both proxy and Shorty.

### Dependency pattern

```python
import hmac

def require_authenticated_user(request: Request, settings: SettingsDep) -> str:
    email = request.headers.get("X-Authenticated-User", "").strip()
    signature = request.headers.get("X-Auth-Signature", "").strip()

    # Basic validation
    if not email or "@" not in email or len(email) > 254:
        raise HTTPException(403, detail="Forbidden")

    # HMAC verification (skip if secret is empty — dev mode)
    if settings.proxy_shared_secret:
        expected = base64.b64encode(
            hmac.new(settings.proxy_shared_secret.encode(), email.encode(), "sha256").digest()
        ).decode()
        if not signature or not hmac.compare_digest(expected, signature):
            raise HTTPException(403, detail="Forbidden")

    return email
```

`hmac.compare_digest` is used (constant-time comparison) to prevent timing attacks.

**Alternative considered**: OIDC token validation with `python-jose` or MSAL directly in Shorty. Rejected for this change — adds library dependencies and Entra ID config complexity that belongs in a future phase. The HMAC approach keeps the backend auth-agnostic while adding defense in depth.

### Header contract

Two headers:
| Header | Value | Required |
|--------|-------|----------|
| `X-Authenticated-User` | Verified email from proxy | Yes |
| `X-Auth-Signature` | `base64(HMAC-SHA256(secret, email))` | Yes (when `PROXY_SHARED_SECRET` is set) |

Email format validation is intentionally minimal (`@` present, ≤ 254 chars) — the proxy already validated the email against Entra ID.

### `created_by` column: nullable `VARCHAR(254)`

The column is nullable for two reasons:
1. Backward compatibility with existing rows that predate auth.
2. Tests that use direct DB inserts without going through the HTTP layer.

```python
created_by: Mapped[str | None] = mapped_column(String(254), nullable=True, default=None)
```

**Alternative considered**: `NOT NULL` with a default sentinel like `"system"`. Rejected because it conflates "authenticated user" with "migration default" — auditing accuracy suffers.

### No auth on `GET /{short_code}`

The redirect endpoint remains public. This is intentional: corporate users share short links with external recipients who must be able to follow them without authentication.

## Risks / Trade-offs

- **[HMAC secret leakage]** If `PROXY_SHARED_SECRET` is exposed (e.g., committed to git, leaked in logs), an attacker can forge signatures. → Mitigation: store in `.env` (already gitignored), never log the secret. Rotate periodically.
- **[Secret unset in production]** If `PROXY_SHARED_SECRET` is empty, HMAC verification is skipped entirely — equivalent to the pre-hardened design. → Mitigation: the secret MUST be documented as a critical production requirement. Consider a startup check that warns if secret is empty.
- **[`created_by` nullable]** Business logic must handle `None` gracefully (existing rows, test fixtures). → Mitigation: the dependency always returns a string; `None` only appears in direct DB inserts or pre-auth rows.
- **[No user entity table]** The email is stored as a plain string in `urls.created_by`. No `users` table, no FK. → Trade-off: simpler, but no referential integrity. Acceptable for MVP since there is no user-facing profile or settings page yet. A future `users` table can backfill from `urls.created_by`.

## Migration Plan

1. Generate Alembic revision for `created_by` column: `alembic revision --autogenerate -m "add created_by to urls"`.
2. Apply migration: `alembic upgrade head`.
3. Existing rows get `NULL` for `created_by` — no data loss.
4. Rollback: `alembic downgrade -1` drops the column.

## Open Questions

- **Should the email be validated against a domain allowlist (e.g., `@empresa.com`)?** Not needed for the initial implementation — the proxy already restricts access to Entra ID users from the configured tenant.
- **Should `created_by` be indexed?** Not yet — querying by creator is a future management feature. Index can be added when needed.
